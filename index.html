Generative AI

From GPT to Production — Building Real Apps with Generative AI

Sumeet Shah June 2026 8 min read ASP.NET Core · C# · OpenAI API
GenAI × .NET

Everyone's played with ChatGPT. Everyone's seen the demos. But there's a significant gap between prompting a chatbot and shipping a production feature powered by a large language model. I've been on both sides of that gap — and this article is about what I learned crossing it.

Over the past year, working at the intersection of enterprise .NET development and emerging AI capabilities, I've integrated LLM-powered features into backend systems that serve real users. Here's what actually matters.

Why most GenAI demos fail in production

Demo code is optimistic. It assumes the model always returns well-formed JSON, that latency is acceptable, that the same prompt works every time. Production systems are none of those things.

The three failure modes I encountered most often:

  • Non-determinism at scale — the same prompt can return structurally different responses. Your parser breaks on Tuesday.
  • Latency spikes — a 2-second average with 15-second tail latency is a UX nightmare when it's blocking a user action.
  • Cost surprises — token usage at demo scale (10 requests/day) doesn't predict production scale (50,000 requests/day).

The architecture that actually works

After several iterations, I settled on a pattern I call Guard-Generate-Validate for LLM integration in .NET backends.

1. Guard — protect your system from the model

Treat the LLM like an untrusted external API. Every response should go through a validation layer before touching your database or business logic.

public async Task<PolicySummary?> GenerateSummaryAsync(string policyText)
{{
    var raw = await _llmClient.CompleteAsync(BuildPrompt(policyText));
    
    if (!TryParseSummary(raw, out var summary))
    {{
        _logger.LogWarning("LLM returned unparseable response, falling back");
        return await _fallbackService.GetSummaryAsync(policyText);
    }}
    
    return summary;
}}

2. Generate — engineer your prompt like code

Prompt engineering is real engineering. I treat prompts as versioned, tested artifacts — not strings concatenated at runtime. A few principles that made a measurable difference:

  • Specify the output format explicitly in the system prompt. Ask for JSON with a defined schema.
  • Include negative examples — "do not include preamble text before the JSON".
  • Use temperature=0 for structured output tasks. Save creativity for summarisation and generation tasks where variation is acceptable.

3. Validate — never trust the model's output directly

JSON schema validation, semantic checks, and range guards before any LLM output reaches your data layer. I use FluentValidation for this — the same library I use everywhere else in the stack.

"The model doesn't know what your application needs. Only your code does."

Practical patterns for ASP.NET Core

Async streaming for better UX

If your LLM call takes 4 seconds, don't make the user wait for the whole response. Stream it. ASP.NET Core supports server-sent events natively — combine it with OpenAI's streaming API and you get a chat-like experience in your own UI without WebSockets complexity.

Caching semantically similar requests

Exact-match caching barely works for LLM inputs — users rephrase the same question. Semantic caching (embedding the input, finding vector-similar cached results) can reduce LLM API calls by 30–60% for common question patterns. I used Azure AI Search for this.

Observability is non-negotiable

Log the full prompt, the full response, latency, token count, and model version for every production call. When something breaks — and it will — you need the complete picture. I pipe this to Splunk with a custom LLM telemetry middleware.

What I'd tell myself 12 months ago

GenAI in production is a software engineering problem first, and an AI problem second. The models are impressive. The hard part is wrapping them in the reliability, observability, and testability that enterprise systems demand. If you can build resilient APIs, you can build resilient AI features — the discipline is the same.

The engineers who will build the most valuable AI-powered systems aren't the ones who know the most about transformer architecture. They're the ones who know how to ship production software — and who learn to work with LLMs the same way they work with any powerful, imperfect dependency.

Agentic AI & .NET — Building Autonomous Backend Workflows

Sumeet Shah May 2026 10 min read ASP.NET Core · C# · Semantic Kernel
Agentic × .NET

Generative AI responds to prompts. Agentic AI acts on goals. That distinction sounds subtle, but it fundamentally changes how you architect the systems around it — and what becomes possible.

A GenAI feature says: "Here's a summary of this document." An agentic system says: "I need to process this claim — let me check the policy database, calculate the payout, flag anomalies, draft the notification, and submit it for approval." It plans, executes, and adapts.

I've been working with agentic patterns in .NET backends since early 2025. Here's what the architecture actually looks like.

What makes a system "agentic"?

Three properties distinguish an agentic system from a simple LLM integration:

  • Tool use — the agent can call functions, APIs, and services autonomously to gather information or take action.
  • Multi-step reasoning — it breaks down complex goals into subtasks and executes them in sequence, with each step informing the next.
  • Feedback loops — it can observe the results of its actions and adjust its plan accordingly.

Microsoft Semantic Kernel — the .NET-native choice

For .NET developers, Semantic Kernel is the most mature framework for building agentic systems. It gives you plugins (tools the agent can call), planners (the reasoning layer), and memory (context persistence) — all wired into familiar .NET patterns.

var kernel = Kernel.CreateBuilder()
    .AddAzureOpenAIChatCompletion(deploymentName, endpoint, apiKey)
    .Build();

kernel.Plugins.AddFromType<PolicyPlugin>();
kernel.Plugins.AddFromType<NotificationPlugin>();

var result = await kernel.InvokePromptAsync(
    "Process claim {{$claimId}} end-to-end and notify the claimant.",
    new KernelArguments {{ ["claimId"] = claimId }}
);

Designing agent plugins for enterprise systems

Each plugin is a C# class with methods decorated with [KernelFunction]. The agent decides when and how to call them based on the goal. The key design principle: make plugins idempotent and auditable.

In enterprise systems, an agent might call the same function twice if it re-plans. Your plugin must handle that gracefully — database operations should be idempotent, and every action should be logged with enough context to reconstruct what happened and why.

The hardest problem: trust boundaries

When a human submits a form, they're responsible for what they submit. When an agent calls your API, the system is responsible. This creates a new class of security and governance challenges.

Scope constraints

Agents should operate within explicitly defined scopes. Don't give your claims-processing agent access to HR data just because it technically could need it. Apply the principle of least privilege just as you would to a service account.

Human-in-the-loop checkpoints

For high-stakes actions — sending a large payment, modifying a live policy, triggering an external API — require explicit human approval before the agent proceeds. I implement this as an async approval workflow: the agent pauses, creates an approval request, and resumes when a human confirms.

"Autonomous doesn't mean unsupervised. The best agentic systems know when to ask."

Full audit trails

Every decision the agent makes — which tool it chose to call, what reasoning it used, what the tool returned — should be logged. When something goes wrong, and eventually it will, you need to understand the agent's decision chain completely.

A real pattern: the claim processing agent

At Accenture, I worked on insurance policy workflows. An agentic system for claim processing looks like this:

  1. Intake — agent receives a claim submission, calls GetPolicyDetails(policyId) to fetch coverage.
  2. Validation — calls ValidateClaim(claimData, policy) to check eligibility and flag anomalies.
  3. Decision — for straightforward claims, calls ApproveClaim() or FlagForReview() based on rules.
  4. Notification — calls SendClaimantNotification() with a generated, personalised message.
  5. Audit — logs the full decision chain to the audit system.

What previously required 3 microservices and a message queue can be expressed as a goal for an agent — with the agent handling orchestration, error recovery, and retry logic autonomously.

Where agentic AI actually delivers value

Not every process benefits from an agent. The sweet spot is workflows that are complex enough that hardcoded orchestration is brittle, but structured enough that the agent can be given clear success criteria.

Document processing, multi-step data enrichment, cross-system reconciliation, intelligent notification routing — these are the use cases where agentic systems consistently outperform both rigid automation and purely human processes.

The backend engineers who understand both the distributed systems fundamentals and the new agentic patterns will build the most impactful systems in the next five years. The foundation is the same — clean architecture, reliable APIs, thoughtful data models. What changes is the orchestration layer above it.
